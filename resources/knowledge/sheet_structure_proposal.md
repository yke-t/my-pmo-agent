# Google Sheets構造改善提案（PMO視点）

## 🎯 改善の目的

15社のベンダー管理において、**誰が・何を・いつまでに**を明確にし、リスクの早期検出と意思決定の迅速化を実現します。

---

## 📋 Issue Log（課題管理）

### 現状の問題点

| 欠落項目 | 影響 |
|---------|------|
| **優先度** | 緊急タスクの見落とし |
| **期限** | 期限超過の検知不可 |
| **ベンダー名** | 15社の責任分界が不明確 |
| **影響範囲** | SIT全体への影響判断不可 |

### 改善案（追加カラム）

```
ID | 起票日 | カテゴリ | 内容 | ベンダー名 | 担当者 | 優先度 | 期限 | ステータス | 影響範囲 | 更新日
```

**追加カラムの定義**:

| カラム名 | 型 | 選択肢/ルール | PMO観点での意義 |
|---------|---|-------------|---------------|
| **ベンダー名** | テキスト | （15社のリスト） | 責任の所在を明確化 |
| **優先度** | プルダウン | `緊急`/`高`/`中`/`低` | トリアージの根拠 |
| **期限** | 日付 | YYYY-MM-DD | SLA管理・アラート基準 |
| **影響範囲** | プルダウン | `全体`/`特定ベンダー`/`限定的` | エスカレーション判断 |
| **更新日** | 日付 | 自動更新（Apps Script） | 放置タスク検知 |

## 📅 Schedule（進捗管理）

### 現状の問題点

| 欠落項目 | 影響 |
|---------|------|
| **担当者** | アサインメント不明確 |
| **ステータス** | 進捗率だけでは「停滞」を検知不可 |
| **依存関係** | クリティカルパス不明 |

### 改善案（追加カラム）

```
ID | タスク | ベンダー名 | 担当者 | 開始予定 | 終了予定 | ステータス | 進捗率 | 依存タスクID | クリティカルパス | メモ
```

**追加カラムの定義**:

| カラム名 | 型 | 選択肢/ルール | PMO観点での意義 |
|---------|---|-------------|---------------|
| **ID** | 数値 | 自動連番 | Issue Logとの紐付け |
| **ベンダー名** | テキスト | （15社のリスト） | ベンダー別進捗集計 |
| **担当者** | テキスト | 実名 | アカウンタビリティ確保 |
| **ステータス** | プルダウン | `未着手`/`進行中`/`停滞`/`完了`/`保留` | 「停滞」の可視化 |
| **依存タスクID** | テキスト | カンマ区切り（例: `1,3,5`） | 依存関係の追跡 |
| **クリティカルパス** | チェックボックス | TRUE/FALSE | 最優先タスクの識別 |

## 🚨 Gemini AIとの連携強化ポイント

### 改善後のAI活用例

**Before（現状）**:
```
/ask 課題数は？
→ 単純カウントのみ
```

**After（改善後）**:
```
/ask 緊急課題は？
→ 「優先度=緊急」かつ「期限3日以内」を抽出
→ 「担当: 鈴木さん、ベンダーA、SIT環境構築遅延」など具体的に提示

/risk-alert
→ 「停滞ステータス7日以上のタスク（クリティカルパス）」を自動検出
→ 「ベンダーA: タスクID=12（API連携テスト）が停滞中。依存タスク3件に影響」
```
